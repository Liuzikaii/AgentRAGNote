### Embedding模型

#### BGE

**BGE 模型量化对比（核心指标）**

| **模型**          | **向量维度** | **模型大小** | **MTEB平均分\*** | **单条embedding耗时（CPU）** | **QPS（CPU）** | **显存/内存占用** |
| ----------------- | ------------ | ------------ | ---------------- | ---------------------------- | -------------- | ----------------- |
| bge-small-en-v1.5 | 384          | ~120MB       | ~62–64           | ~3–8 ms                      | ~120–300/s     | ~0.5–1GB          |
| bge-base-en-v1.5  | 768          | ~420MB       | ~64–66           | ~8–20 ms                     | ~50–120/s      | ~1–2GB            |
| bge-large-en-v1.5 | 1024         | ~1.3GB       | ~66–69           | ~20–60 ms                    | ~10–40/s       | ~3–6GB            |

**指标简单解释一下：**

- **MTEB（精度）**：差2分能感觉到差距，差4分算质变。从 small 换到 base，命中率大概从 70-80% 提升到 75-85%，体感不明显。真要明显提升得换 large（命中率能到 80-90%）。
- **延迟和 QPS（CPU单机环境）**：做高并发场景这个很关键。small 几乎没延迟；base 还算能接受；large 跑单条就有几十毫秒，并发一上来直接卡顿。
- **资源占用**：small 随便跑；base 需要稍微调优；large 强烈建议上 GPU，不适合用 CPU 硬扛高并发。

**真实的工程结论：**

在 RAG 系统里，embedding 模型其实不是决定最终效果的唯一因素。

实际对效果的影响排序如下：

```
RAG 调优核心法则：效果影响因子排序 (Chunk 策略 > Reranker > Embedding 模型大小)

1. 为什么 Chunk 策略排第一？（决定系统的天花板）
   计算机科学里有一句老话：Garbage in, garbage out。
   Embedding 模型再强大，也只是个负责把文本翻译成向量的“翻译官”。
   - 切分太碎（丢失上下文）：按短句硬切分后，代词和主语脱离，提问时直接导致检索失败。
   - 切分太大（噪声稀释）：大段落包含多个独立知识点，提问单一知识点时，整体向量相似度会被无关内容稀释。
   工程现实：优秀的切分策略能直接确保喂给大模型的片段是完整且干净的。这是哪怕动用千亿参数模型也无法替你做的事情。

2. 为什么 Reranker 排第二？（打破双塔模型的精度瓶颈）
   这涉及到底层架构的差异：
   - 初排模型（Embedding）：问题和文档各自独立计算向量，最后比对距离。速度极快，但计算时两者没有“见过”彼此，无法进行深度语境比对。
   - 重排模型（Reranker）：把问题和召回的文档拼在一起直接输入给模型，让模型逐词对比细节。
   工程现实：用一个几十兆的 Reranker 模型做二次重排，带来的准确度提升，往往能吊打你把 Embedding 模型从 100M 升级到 2GB 的效果，在召回层面属于降维打击。

3. 为什么 Embedding 模型大小排最后？（边际收益严重递减）
   并不是说 Embedding 不重要，而是目前主流开源基础模型的能力已经溢出了。
   - 及格线极高：100MB 出头的轻量级模型（如 bge-small），语义表征能力已足够应付绝大多数业务场景。
   - 边际效应骤减：把模型体积放大十倍（换成 large 版本），获取的可能只是从 80 分到 85 分的微小提升。
   - 工程代价极大：盲目推高向量维度会导致内存占用翻倍，检索计算量飙升，单机并发（QPS）直接雪崩。

工程总结与行动指南
   核心认知：
   Chunk 决定了你递给大模型的“砖块”好不好；
   Reranker 决定了你能否从一堆砖块里精准挑出最对症的那块；
   Embedding 大小，只是决定了这块砖的表面打磨得有多光滑。
   执行建议：
   - 先通流程：初期闭眼选最轻量的 Embedding 模型（如 bge-small）先把架构跑通，扛住并发。
   - 死磕数据：把 70% 的精力花在清洗脏数据和打磨 Chunk 切分算法上（建议控制在 200~500 tokens）。
   - 低成本提效：遇到召回精度瓶颈时，优先在架构里接入 Reranker 模块。
   - 慎用大模型：如果没有独立的 GPU 算力支撑，且不是极低并发的场景，不要轻易上 Large 级别的模型。
```

**选型建议：**

如果项目赶时间，且未来要兼顾高并发和分布式，建议按这个路线走：

1. **第一阶段**：直接上 bge-small。性能最好，工程落地的首选。
2. **第二阶段**：bge-small + bge-reranker-base。加上 reranker 之后，效果能直接盖过单跑一个 large 的效果，而且不会把系统拖垮。
3. **第三阶段**：等你遇到低 QPS 但需要极致精度的场景，再去考虑换 large 模型。

**比换模型更重要的几个优化点：**

- **chunk size**：控制在 200~500 tokens 最佳。切分算法要弄好，不然语义断裂了换什么模型都没用。
- **topK**：一般设为 3~5 就行。
- **reranker**：核心组件。用 small 加上 reranker，性价比和效果直接拉满。
